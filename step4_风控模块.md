基于现有仓库与共用约束，请开发“风控模块”。

目标：
实现一个独立、可测试、可配置的风险控制层，位于“策略输出 / 目标仓位 / 订单请求”和“执行层”之间。

设计原则：
- 风控必须独立于策略层
- 风控必须独立于执行实现
- 风控结果必须可解释、可审计
- 风控既可以用于回测，也可以用于模拟执行

必须实现的能力：

一、风控主引擎
请实现：
- RiskEngine
- 风控配置对象
- RiskDecision 或等价结构
- 风控审计日志结构

风控输入至少要支持：
- PortfolioSnapshot
- 当前市场状态（最小可用）
- TargetPosition 或 OrderRequest
- 可选的历史净值 / 当前回撤信息

风控输出至少要支持：
- approve（通过）
- modify / clip（裁剪）
- reject（拒绝）
并附带 reason / rule_name / metadata

二、规则抽象
请设计统一规则基类，例如：
- BaseRiskRule

并支持将多条规则组合执行，规则顺序要清晰，冲突时要有明确优先级。

三、至少实现以下 6 类规则
1. 单标的最大仓位限制
2. 组合最大总仓位 / 最大杠杆限制
3. 单日最大换手限制
4. 黑名单 / 白名单
5. 停牌 / 不可交易过滤
6. 最大回撤熔断

如果你的架构更合理，也可以再补充：
- 行业集中度限制
- 最小成交额 / 流动性过滤
- 单笔订单数量限制

四、风控与回测衔接
请让回测模块可以选择接入 RiskEngine：
- 策略产生目标仓位或订单请求
- 先进入风控
- 再进入回测内部撮合 / 执行流程

五、日志与审计
每一条被修改或被拒绝的请求，都要能看到：
- 是哪条规则触发
- 原始值是什么
- 修改后是什么
- 拒绝原因是什么

六、测试
至少覆盖：
- 仓位超限被裁剪
- 黑名单标的被拒绝
- 达到最大回撤后被熔断
- 不可交易标的被过滤
- 多条规则组合时行为可预测
- 风控接入回测链路后可运行

七、文档与示例
请提供：
- 风控接入示例
- 风控规则配置示例
- 新增一条风控规则的方法说明
- 风控决策输出示例

实现约束：
- 不要把规则分散写在 strategy / backtest / execution 各处
- 不要只返回布尔值，要返回可审计结果
- 配置要尽量清晰
- 允许后续继续增加更多规则
- 如有必要，可以对回测模块做最小兼容性改造

交付验收标准：
- risk 模块可独立测试
- 回测链路可插入 risk engine
- 能看到每笔请求被通过、裁剪、拒绝的原因
- pytest 可以通过
- 文档足以支持后续新增规则

完成后请输出：
1. 改动文件清单
2. 风控引擎与规则优先级说明
3. 如何在回测链路中启用风控
4. 示例规则配置
5. 后续最容易扩展的风控点
